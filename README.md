
## 1. HTTP(HyperText Transfer Protocol) 통신 상세

HTTP는 웹 상에서 클라이언트와 서버가 자원(HTML, 이미지, 데이터 등)을 주고받기 위해 사용하는 **응용 계층 프로토콜**입니다. 단순한 텍스트 전송을 넘어, 오늘날 현대적인 웹 애플리케이션의 뼈대를 이룹니다.



### 1) 주요 특징 및 원칙
*   **Stateless (무상태성):** 서버는 클라이언트의 이전 요청 상태를 저장하지 않습니다. 
    *   *장점:* 서버의 확장성이 높음 (어떤 서버가 응답해도 상관없음).
    *   *보완:* 로그인 상태 유지 등을 위해 **Cookie, Session, JWT(Token)** 등을 활용합니다.
*   **Connectionless (비연결성):** 요청과 응답이 끝나면 연결을 즉시 해제하여 서버 자원을 효율적으로 관리합니다.
    *   *보완:* 매번 연결을 맺는 오버헤드를 줄이기 위해 **Keep-Alive** 기능을 주로 사용합니다.

### 2) HTTP 메세지 구조
HTTP 요청과 응답은 크게 세 부분으로 나뉩니다.
1.  **Start Line:** 요청 시에는 `Method / 경로 / 버전`, 응답 시에는 `버전 / 상태코드 / 상태문구`가 들어갑니다.
2.  **Headers:** 통신에 필요한 부가 정보(호스트 주소, 브라우저 정보, 콘텐츠 타입, 인증 정보 등)를 담습니다.
3.  **Body:** 실제 전송할 데이터(JSON, HTML, 이미지 파일 등)가 포함됩니다. (GET 요청은 보통 Body가 비어있습니다.)

### 3) 주요 HTTP 메서드 (Methods)
용도에 맞는 메서드 사용은 **RESTful한 API** 설계의 기본입니다.
*   `GET`: 서버로부터 자원을 **조회**할 때 사용 (데이터 변경 없음)
*   `POST`: 서버에 데이터를 전달하여 자원을 **생성**할 때 사용
*   `PUT`: 자원을 전체 **수정**하거나 교체할 때 사용
*   `PATCH`: 자원의 **일부만 수정**할 때 사용
*   `DELETE`: 특정 자원을 **삭제**할 때 사용

### 4) HTTP 상태 코드 (Status Codes)
서버는 응답 시 숫자로 된 상태 코드를 통해 처리 결과를 알립니다.
*   `2xx (Success)`: 요청 성공 (예: 200 OK, 201 Created)
*   `3xx (Redirection)`: 요청을 완료하려면 추가 작업(페이지 이동 등)이 필요함
*   `4xx (Client Error)`: 클라이언트의 요청이 잘못됨 (예: 404 Not Found, 401 Unauthorized)
*   `5xx (Server Error)`: 서버 측 오류로 처리에 실패함 (예: 500 Internal Server Error)

### 5) HTTP 버전의 진화
*   **HTTP/1.1:** 가장 널리 쓰이는 버전. Keep-Alive로 연결을 재사용하지만, 여러 파일을 받을 때 순서대로 기다려야 하는 HOLB(Head of Line Blocking) 문제가 있음.
*   **HTTP/2.0:** 한 번의 연결로 여러 요청을 동시에 처리(Multiplexing)하여 속도를 획기적으로 개선.
*   **HTTP/3.0:** TCP 대신 **UDP(QUIC)** 프로토콜을 사용하여 연결 설정 시간을 단축하고 패킷 손실에 더 강해짐.

## 2. 브라우저에 URL을 입력했을 때의 전체 과정
사용자가 브라우저 주소창에 URL을 입력하면 다음과 같은 복잡한 단계를 거쳐 화면이 나타납니다.

### Step 1: DNS Lookup (주소 찾기)
브라우저는 도메인 이름(예: `www.google.com`)을 컴퓨터가 이해할 수 있는 **IP 주소**로 변환하기 위해 DNS 서버에 요청을 보냅니다.

### Step 2: TCP 3-Way Handshake (연결 수립)
데이터를 전송하기 전, 클라이언트와 서버 간의 신뢰성 있는 연결을 위해 세 번의 확인 과정을 거칩니다. (SYN -> SYN-ACK -> ACK)

### Step 3: HTTP Request & Response (데이터 요청 및 응답)
1.  **Request:** 브라우저가 서버에 HTML, 이미지 등의 데이터를 달라고 요청을 보냅니다.
2.  **Response:** 서버는 요청을 확인하고 해당 데이터를 응답 메시지에 담아 보냅니다.

### Step 4: Browser Rendering (화면 그리기)
받아온 데이터를 브라우저 엔진이 처리하여 사용자에게 보여줍니다.
1.  **Parsing:** HTML을 읽어 `DOM 트리`를, CSS를 읽어 `CSSOM 트리`를 만듭니다.
2.  **Render Tree:** 위 두 트리를 합쳐 화면에 실제로 보일 `Render 트리`를 생성합니다.
3.  **Layout & Paint:** 각 요소의 위치를 계산하고 화면에 실제 픽셀을 채웁니다.
